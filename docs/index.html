§ 01 · WHY Šváb Pi

Why Šváb Pi? 왜 Šváb Pi인가요?

●
Fully Open Source 완전한 오픈소스

Every line is on GitHub. No hidden prompts, no secret system instructions, no obfuscated behavior. Tools, event hooks, skill injections, and agent prompts are all plain TypeScript and Markdown.

모든 코드가 GitHub에 공개되어 있습니다. 숨겨진 프롬프트, 비밀 시스템 지시어, 난독화된 동작 없음. 도구, 이벤트 훅, 스킬 주입, 에이전트 프롬프트가 모두 일반 TypeScript와 Markdown으로 작성되어 있습니다.

★
Observable 옵저버빌리티

The footer displays real-time metrics every session: prompt cache hit rate, context usage bar, active tools, branch, and model. See exactly how your context is being utilized.

푸터에 매 세션 실시간 메트릭이 표시됩니다: 프롬프트 캐시 적중률, 컨텍스트 사용량 바, 활성 도구, 브랜치, 모델. 컨텍스트가 어떻게 사용되는지 정확히 확인하세요.

■
Transparent by Design 디자인부터 투명하게

No magic. Read the source and see exactly what the agent does. Skill rules are plain Markdown. Agent prompts are plain text. Every event hook is auditable.

마법 없음. 소스를 읽고 에이전트가 정확히 무엇을 하는지 확인하세요. 스킬 규칙은 일반 Markdown, 에이전트 프롬프트는 일반 텍스트, 모든 이벤트 훅이 감사 가능합니다.

▶
Simple Over Clever 화려함보다 단순함

We prefer straightforward solutions over elaborate orchestration. Fewer moving parts, fewer failure modes. See our contributing guidelines.

복잡한 오케스트레이션보다 단순한 해결책을 선호합니다. 움직이는 부분이 적을수록 실패도 적습니다. 기여 가이드라인을 참조하세요.

  1. 01

    /setup — Configure Settings — 설정 구성

    Run once after installation to apply recommended settings. Sets quietStartup: true so the extension's custom Šváb Pi banner replaces the default startup listing.

    설치 후 한 번 실행하여 권장 설정을 적용합니다. 익스텐션의 커스텀 Šváb Pi 배너가 기본 시작 목록을 대체하도록 quietStartup: true를 설정합니다.

  2. 02

    /clarify — Resolve Ambiguity — 모호성 해결

    The agent asks dynamic, context-aware questions one at a time, using codebase exploration only when implementation or uncertain technical context makes it necessary. Ends with a structured Goal Contract that defines scope, constraints, success criteria, and evidence requirements.

    에이전트가 동적이고 문맥에 맞는 질문을 하나씩 묻고, 구현이나 불확실한 기술 맥락 때문에 필요할 때만 코드베이스를 탐색합니다. 범위, 제약, 성공 기준, 증거 요구사항을 정의하는 구조화된 Goal Contract로 끝납니다.

  3. 03

    /goal — Activate Durable Execution — durable 실행 활성화

    Starts or continues a durable runtime that tracks goals, subgoals, todos, evidence, blockers, verifier receipts, and automatic continuation. Completion requires an independent verifier PASS.

    목표, 하위 목표, todo, 증거, 차단 요소, 검증 영수증, 자동 이어가기를 추적하는 durable runtime을 시작하거나 이어갑니다. 완료에는 독립 verifier PASS가 필요합니다.

  4. 04

    /goal subgoal — Subgoal Queueing — 하위 목표 큐잉

    For complex multi-step tasks. Queue subgoals with dependencies and required evidence, then let the goal runtime advance after verifier-guarded completion.

    복잡한 다단계 작업을 위해. 의존성과 필요한 증거가 있는 하위 목표를 큐에 넣고, verifier-guarded 완료 후 goal runtime이 진행하게 합니다.

  5. 05

    /loop — Session Loop Scheduler — 세션 루프 스케줄러

    Schedule recurring prompts at fixed intervals. Cron-style timing — fires on schedule regardless of execution state. Supports 5s, 10m, 2h, 1d formats. Up to 100 concurrent jobs with per-job error isolation and automatic cleanup on session shutdown.

    고정 간격으로 반복 프롬프트를 스케줄링합니다. Cron 스타일 타이밍 — 실행 상태와 관계없이 스케줄대로 실행됩니다. 5s, 10m, 2h, 1d 형식을 지원합니다. 최대 100개 동시 작업, 작업별 에러 격리, 세션 종료 시 자동 정리.

💡 Tip: The ask_user_question tool is always available — the agent will ask clarifying questions autonomously whenever it detects ambiguity, even outside /clarify mode.

💡 팁: ask_user_question 도구는 항상 사용 가능합니다 — /clarify 모드 외에서도 모호성을 감지하면 자동으로 질문합니다.

§ 04 · FAQ

FAQ

What is the pi coding agent? pi 코딩 에이전트란 무엇인가요?

pi is a terminal-based AI coding agent that helps you read, write, and debug code directly from your terminal. It supports extensions, custom tools, themes, and multiple AI providers. GitHub →

pi는 터미널에서 직접 코드를 읽고, 쓰고, 디버깅하는 터미널 기반 AI 코딩 에이전트입니다. 익스텐션, 커스텀 도구, 테마 및 여러 AI 프로바이더를 지원합니다. GitHub →

Do I need to install the engineering-discipline skills separately? 엔지니어링 디사이플린 스킬을 별도로 설치해야 하나요?

No. All engineering discipline skills are bundled with this extension. Just install the extension and run /setup — everything is ready out of the box.

아니요. 모든 엔지니어링 디사이플린 스킬이 이 익스텐션에 번들로 포함되어 있습니다. 익스텐션을 설치하고 /setup을 실행하세요 — 모든 것이 즉시 사용 가능합니다.

How do the subagent modes work? 서브에이전트 모드는 어떻게 작동하나요?

Single: One-off tasks sent to a specific agent.
Parallel: Dispatch multiple agents at once (max 12 tasks, 10 concurrent). Each runs independently.
Chain: Sequential pipeline where each step receives the previous step's output via {previous} placeholder.
All modes include cycle detection, depth limits (max 3), and concurrency control.

단일: 특정 에이전트에 보내는 일회성 작업.
병렬: 여러 에이전트를 한 번에 디스패치(최대 12개 작업, 10개 동시). 각각 독립적으로 실행.
체인: 각 단계가 이전 단계의 출력을 {previous} 플레이스홀더로 받는 순차 파이프라인.
모든 모드에 사이클 감지, 깊이 제한(최대 3), 동시성 제어가 포함됩니다.

How does engineering discipline stay enforced? 엔지니어링 디사이플린은 어떻게 유지되나요?

The harness injects focused implementation guardrails into code-writing agents: read before writing, keep changes scoped, verify assumptions, and record evidence before completion.

하네스는 코드 작성 에이전트에 집중된 구현 가드레일을 주입합니다. 쓰기 전에 읽고, 변경 범위를 지키며, 가정을 검증하고, 완료 전에 증거를 기록합니다.

How does the FFF search engine change my workflow? FFF 검색 엔진은 워크플로를 어떻게 바꾸나요?

It overrides the built-in find and grep tools with a native FFF-backed implementation and adds multi_grep. In both mode it also replaces @ file autocomplete suggestions. Use /fff-health to inspect engine status and /fff-mode tools-only if you only want the tool overrides.

기본 find / grep 도구를 네이티브 FFF 기반 구현으로 교체하고 multi_grep를 추가합니다. both 모드에서는 @ 파일 자동완성도 대체합니다. 엔진 상태는 /fff-health로 확인하고, 도구 override만 원하면 /fff-mode tools-only를 사용하세요.

How do I handle complex multi-step goals? 복잡한 다단계 목표는 어떻게 처리하나요?

Use /clarify to produce a Goal Contract, then /goal to create the durable runtime entry. Add subgoals when the work needs decomposition.

/clarify로 Goal Contract를 만든 다음 /goal로 durable runtime 항목을 만드세요. 작업 분해가 필요하면 하위 목표를 추가하세요.
